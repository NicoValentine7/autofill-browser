# Vault Key recovery uses passphrase wrapping

Autofill Browser recovers a Vault Key on another device by storing a Vault Recovery Package in D1: the client wraps the Vault Key with a high-entropy client-generated Recovery Phrase using PBKDF2-SHA256 at 600k iterations and AES-GCM, then syncs only that encrypted package. The AES-GCM operation authenticates the recovery package metadata as AAD, including schema, algorithm, KDF params, salt, IV, keyId, vaultId, and createdAt. Restore rejects tampered metadata and requires the unwrapped Vault Key keyId to match the package keyId.

Each Secure Vault root carries a `vaultId`, `activeKeyId`, and encrypted key-check canary. Recovery validates the package `vaultId` against the current vault and validates the unwrapped key against the canary before saving the key into `chrome.storage.session`. Legacy plaintext Vault Keys found in `chrome.storage.local` are migrated into session storage and then cleared from local storage. This preserves the Zero-Knowledge Vault boundary because the Worker can authenticate and move the package, but it never receives the Recovery Phrase or plaintext Vault Key. The Worker rejects inbound sync payloads containing `secureVaultKey`; an admin scrub endpoint exists to remove legacy `secureVaultKey` material from current/history sync rows.

For the release-minded v1, Secure Vault values are not filled during page load, navigation, mutation, or settings updates. They are filled only from the popup-triggered manual run. CVV/CVC/CID/security-code fields are blocked from durable learning entirely. The current content script still reads extension storage directly, so `chrome.storage.session` remains accessible to the extension content script until storage access is brokered through the service worker in a later hardening pass.
